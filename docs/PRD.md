# プロダクト要求定義書 (PRD)

## 1. プロダクト概要

### 1.1 プロダクト名
Board Game Algorithm Visualizer（仮称）

### 1.2 ビジョン
ボードゲームにおける探索・学習アルゴリズムの動作を、インタラクティブかつリアルタイムに可視化するWebアプリケーション。アルゴリズムが「どう考えているか」を直感的に理解できる体験を提供する。

**既存ツールとの差別化**: 多くの可視化ツールはゲームとアルゴリズムが1対1で固定されている。本プロダクトはプラグイン設計により「任意のゲーム × 任意のアルゴリズム」を自由に組み合わせて比較・学習できる点が独自の価値である。

### 1.3 対象ユーザー
**プライマリユーザー（MVP）**: アルゴリズムを自習中の学生・エンジニア
- 教科書や講義でMinimax、Alpha-Beta、MCTSなどを学んだが、動作の直感的理解が足りないと感じている
- 自分のペースでステップ実行しながら「なぜこの枝が刈られるのか」を確認したい

**将来のユーザー（Phase 2以降）**:
- ゲームAIに興味がある開発者（自作アルゴリズムの動作確認）
- 探索アルゴリズムを教える教育者（授業での実演）

### 1.4 解決する課題
- 探索アルゴリズムは教科書の静的な図だけでは動作を理解しにくい
- 枝刈りやUCB選択などの挙動は、実際に動く過程を見て初めて腹落ちする
- 既存ツールはゲームとアルゴリズムが固定されており、比較や拡張がしにくい

### 1.5 利用シーン
- 自習: 教科書を読みながら、対応するアルゴリズムの挙動をステップ実行で確認する
- 比較学習: MinimaxとAlpha-Betaを同じ局面で動かし、枝刈りの効果を実感する
- 実験: 「この局面でMCTSは何回シミュレーションすれば正しい手を見つけるか」を試す

## 3. 機能要件

### 3.1 MVP（Phase 1）

#### F1: 盤面表示・操作
- 三目並べ・Kings Valley の盤面をSVGで描画する
- ユーザーがクリックで手を打てる
- 先手・後手を選択できる（ゲーム開始前に選択）
- 現在の手番を表示する
- 勝敗・引き分けを判定し表示する
- ヘッダーのドロップダウンでゲームを切り替えられる（切り替え時に盤面をリセット）

#### F2: アルゴリズム実行
- Minimaxアルゴリズムで最善手を探索する
- Alpha-Betaアルゴリズムで最善手を探索する
- 探索中の各ステップをWebSocketイベントとしてフロントエンドに送信する（イベント仕様は3.4節参照）

#### F3: 探索ツリー可視化
- 探索ツリーをノード・エッジで表示する
- 各ノードに評価値を表示する
- Alpha-Betaで枝刈りされたノードを視覚的に区別する（グレーアウト等）
- ノードクリックでその局面を盤面パネルにプレビュー表示する
- 表示ノード数の上限は500ノードとし、超過時は深さで打ち切りノード折りたたみで対応する
- 折りたたまれたノードはクリックで展開できる

#### F4: 再生コントロール
- ステップ実行: 1イベント単位で進める（ノード展開1つ分）
- 自動再生 / 一時停止
- 再生速度の調整（スライダー: 0.5x〜5x）
- リセット（初期局面に戻す）
- 戻る（1ステップ前の状態に戻す）

#### F5: アルゴリズム切り替え
- ヘッダーからMinimax / Alpha-Betaを切り替えられる
- 切り替え時にツリーをリセットして再探索する

### 3.2 Phase 2

#### F6: MCTS対応
- モンテカルロ木探索を実装する
- ノードサイズを訪問回数に応じて変化させる
- UCB値をノードに表示する
- シミュレーション回数を設定可能にする

#### F7: オセロ対応
- オセロのゲームロジックを追加する
- 8x8盤面をSVGで描画する
- 合法手のハイライト表示

#### F8: 対戦モード
- 人間 vs AI で対戦できる
- AIの思考過程をリアルタイムに可視化する（思考中にツリーが成長する様子を表示）

### 3.3 Phase 3（将来構想）

#### F9: 強化学習の可視化
- 学習済みモデルの評価値を盤面上にヒートマップ表示する
- 学習曲線（報酬の推移）をグラフ表示する

#### F10: アルゴリズム比較
- 2つのアルゴリズムを左右に並べて比較表示する
- 探索ノード数・時間などの統計を対比する

#### F11: 局面エディタ
- 盤面を自由に編集して任意の局面から探索を開始できる

#### F12: 共有機能
- 特定の局面・探索状態をURLで共有できる

### 3.4 WebSocketイベント仕様（フロントエンド・バックエンド間の契約）

探索アルゴリズムは以下のイベントをステップごとに送信する。フロントエンドはこのイベント列を受け取り、ツリーを逐次構築する。

| イベント種別 | 内容 | ペイロード例 |
|---|---|---|
| `node_expanded` | 新しいノードを展開した | `{ id, parent_id, move, state }` |
| `node_evaluated` | ノードの評価値が確定した | `{ id, value }` |
| `node_pruned` | Alpha-Beta枝刈りが発生した | `{ id, alpha, beta }` |
| `search_complete` | 探索が完了した | `{ best_move, total_nodes, elapsed_ms }` |

Phase 2でMCTS追加時に拡張:

| イベント種別 | 内容 | ペイロード例 |
|---|---|---|
| `node_selected` | UCBに基づきノードを選択した | `{ id, ucb_value }` |
| `simulation_done` | ランダムシミュレーションが完了した | `{ id, result }` |
| `backpropagated` | 結果を親ノードに逆伝播した | `{ id, visits, win_rate }` |

## 4. 非機能要件

### 4.1 パフォーマンス
- 三目並べの全探索が5秒以内に完了すること
- ツリー表示は500ノード程度までスムーズに描画できること（60fps維持）
- WebSocket通信の遅延がステップ実行の体験を損なわないこと（イベント送信は100ms以内）

### 4.2 ユーザビリティ
- 初回訪問時に操作方法が直感的にわかること
- ステップ実行で探索の1手1手を追えること
- ツリーのズーム・パンが自然に操作できること

### 4.3 拡張性
- 新しいゲームの追加がGamePluginインターフェースの実装のみで可能なこと
- 新しいアルゴリズムの追加がAlgorithmPluginインターフェースの実装のみで可能なこと

### 4.4 動作環境
- モダンブラウザ（Chrome, Firefox, Edge最新版）
- デスクトップ解像度を想定（レスポンシブ対応はPhase 3以降）
- ローカル環境で動作（デプロイ不要）

### 4.5 エラーハンドリング
- WebSocket切断時: フロントエンドに切断を通知し、自動再接続を試みる（3回まで）
- バックエンド異常終了時: フロントエンドにエラーメッセージを表示し、リセット操作を案内する
- 探索タイムアウト: 30秒以内に探索が完了しない場合、途中結果を返しタイムアウトを通知する

### 4.6 テスト方針
- ゲームロジック（合法手生成、勝敗判定）: ユニットテスト
- アルゴリズム（探索結果の正しさ、イベント送出順）: ユニットテスト
- WebSocket通信: 統合テスト
- フロントエンド: コンポーネント単位のユニットテスト + ツリー描画の手動確認
